## Next.js App Router의 'postings' 폴더 구조 설명

### 1. 'postings' 폴더 구조 개요
`frontend/src/app/postings` 폴더는 Next.js App Router에서 채용 공고와 관련된 라우팅 및 UI를 관리하는 핵심적인 부분입니다. 이 폴더 안에는 다음과 같은 주요 파일 및 폴더가 존재합니다.

*   `frontend/src/app/postings/page.js`: 모든 채용 공고 목록을 표시하는 페이지를 담당합니다.
*   `frontend/src/app/postings/[posting_idx]/page.js`: 특정 채용 공고의 상세 정보를 표시하는 페이지를 담당합니다. 여기서 `[posting_idx]`는 URL에서 동적으로 변하는 값을 나타내는 동적 라우트 세그먼트입니다.
*   `frontend/src/app/postings/[posting_idx]/JobPostingDetailClient.js`: 위 상세 페이지의 클라이언트 측 로직 및 UI를 담당하는 컴포넌트입니다.

### 2. 흐름 (Flow)
사용자가 채용 공고 관련 페이지에 접근할 때의 흐름은 다음과 같습니다.

*   **채용 공고 목록 페이지 (`/postings`)**:
    1.  사용자가 `http://localhost:3000/postings` URL로 접속합니다.
    2.  Next.js는 `frontend/src/app/postings/page.js` 파일을 로드하여 채용 공고 목록을 렌더링합니다.
    3.  이 페이지는 백엔드 API (`/api/search`)를 호출하여 채용 공고 데이터를 가져오고, 이를 기반으로 여러 개의 채용 공고 카드들을 표시합니다.
*   **특정 채용 공고 상세 페이지 (`/postings/123`)**:
    1.  사용자가 목록 페이지에서 특정 채용 공고 카드를 클릭하거나, `http://localhost:3000/postings/123` (예시)와 같은 상세 URL로 직접 접속합니다.
    2.  Next.js는 `frontend/src/app/postings/[posting_idx]/page.js` 파일을 로드합니다.
    3.  `page.js` (서버 컴포넌트)는 URL에서 `posting_idx` 값(예: `123`)을 추출하여 `JobPostingDetailClient.js` (클라이언트 컴포넌트)에 prop으로 전달합니다.
    4.  `JobPostingDetailClient.js`는 전달받은 `posting_idx`를 사용하여 백엔드 API (`/api/job-postings/idx/123`)를 호출하고, 상세 데이터를 가져와 페이지를 렌더링합니다.

### 3. 필요성 및 차이점 (Necessity & Differences)
이러한 구조는 Next.js App Router의 핵심적인 설계 원칙과 밀접하게 관련되어 있으며, 다음과 같은 필요성 및 차이점을 가집니다.

*   **동적 라우팅의 구현**: `[posting_idx]`와 같은 동적 세그먼트 폴더는 URL의 특정 부분이 변수임을 Next.js에 알려줍니다. 이를 통해 수많은 채용 공고 각각에 대해 별도의 파일을 만들 필요 없이 하나의 템플릿으로 상세 페이지를 처리할 수 있습니다.
*   **서버 컴포넌트와 클라이언트 컴포넌트의 분리 (SSR/CSR 최적화)**:
    *   **`page.js` (서버 컴포넌트)**: 기본적으로 서버에서 렌더링됩니다. 이는 초기 페이지 로드 시 사용자에게 빠르게 콘텐츠를 보여주고, 클라이언트로 전송되는 JavaScript 번들 크기를 최소화하여 웹 성능을 향상시킵니다. `params`와 같은 라우트 정보를 서버에서 안전하고 효율적으로 처리할 수 있습니다.
    *   **`JobPostingDetailClient.js` (클라이언트 컴포넌트)**: `use client` 지시어를 통해 클라이언트에서 렌더링됨을 명시합니다. `useState`, `useEffect`와 같은 React Hooks를 사용하여 사용자 인터랙션(버튼 클릭, 데이터 동적 업데이트 등)이 필요한 부분을 담당합니다. 모든 로직을 서버에서 처리할 수 없으므로, 인터랙티브한 부분은 클라이언트 컴포넌트에 위임합니다.
*   **코드의 응집도 및 유지보수성**: 특정 라우트(예: 채용 공고 상세)와 관련된 서버 및 클라이언트 로직을 하나의 폴더(`[posting_idx]`) 안에 함께 배치함으로써, 관련 코드를 쉽게 찾고 관리할 수 있어 코드의 응집도가 높아지고 유지보수성이 향상됩니다.

### 4. 이렇게 해야 하는 이유 (Why this approach is taken)
이러한 구조는 Next.js가 현대 웹 애플리케이션의 성능, 확장성, 개발 편의성을 동시에 잡기 위해 고안한 방식입니다.

*   **최적화된 사용자 경험**: 서버 컴포넌트를 통해 빠른 초기 로딩과 SEO(검색 엔진 최적화) 이점을 얻고, 클라이언트 컴포넌트를 통해 풍부한 인터랙티브 경험을 제공합니다.
*   **개발 효율성**: 파일 시스템 기반 라우팅은 직관적이며, 서버/클라이언트 컴포넌트의 명확한 분리는 개발자가 각 컴포넌트의 역할에 집중할 수 있도록 돕습니다.
*   **Next.js의 강력한 기능 활용**: 이 구조를 따름으로써 Next.js가 제공하는 데이터 페칭, 캐싱, 라우팅 등의 고급 기능을 최대한 활용할 수 있습니다.

결론적으로, 이 폴더 구조는 Next.js App Router의 설계 철학을 반영하며, 복잡해 보일 수 있지만 실제로는 효율적이고 강력한 웹 애플리케이션을 구축하기 위한 최적의 방법 중 하나입니다.
